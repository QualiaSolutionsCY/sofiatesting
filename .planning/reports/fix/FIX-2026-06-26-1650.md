# Fix Report - 2026-06-26 16:50

**Symptom:** Emailing an invoice delivers TWO emails — one from "CSC Zyprus Property Group" (subject "Invoice 11445 — CSC Zyprus Property Group") and one from "SOPHIA" (subject "Fawzi Goussous — № 11445"). User wants only ONE (the PDF is enough).
**Mode:** quick (1 file)
**Outcome:** fixed

## Feedback Loop
- Initial inspection: traced both senders. "CSC Zyprus" = `sendInvoiceEmailAction` (`documents.ts:444-446`, the proper invoice email). "SOPHIA" = the bot's generic `sendEmail` tool (`tools/handlers/email.ts:131`, from "SOPHIA <sophia@zyprus.com>").
- Final passing check: prompt now forbids the generic sendEmail for invoice documents; verified at edge deploy.

## Root Cause
`supabase/functions/sophia-bot/prompts/behaviors/invoicing.ts` — the invoicing prompt had no rule scoping invoice emails to the `email_invoice` intent, while `prompts/core/safety-rules.ts:34` says "ALWAYS call sendEmail when user wants something emailed." So for an invoice-email request Sophia fired BOTH: `manageInvoice email_invoice` (→ official "CSC Zyprus" email with PDF) AND the generic `sendEmail` (→ duplicate "SOPHIA" email).

## Patch
- File changed: `supabase/functions/sophia-bot/prompts/behaviors/invoicing.ts` (1)
- Added an `email_invoice` intent mapping plus an explicit rule: emailing an invoice / receipt / credit note goes through `email_invoice` ONLY; NEVER also call the generic `sendEmail` tool (it delivers a duplicate). One email with the PDF is enough. `invoicing` is in FILE_OVERRIDE_KEYS, so the file is the live source.

## Verification
- `supabase functions deploy sophia-bot` — compiles the edge fn (real validator for Deno files; tsc doesn't cover them).
- Live WhatsApp retest — PENDING (user): ask Sophia to email an invoice, confirm only ONE email arrives (the "CSC Zyprus" one with the PDF).
