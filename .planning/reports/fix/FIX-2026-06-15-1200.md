# Fix Report - 2026-06-15 12:00

**Symptom:** Live test results — Test 1 (REMU `listing-17069`): "None of the images could be uploaded — Not an image (text/html)". Test 2 (Altamira `pr43514`): "the only missing field is the price" though the page clearly shows €850,000.
**Mode:** general (bank-portal extraction)
**Outcome:** fixed + LIVE (both in services/portal-scraper.ts).

## Root Cause
- **REMU images** — REMU is a JS SPA; gallery photos are served by `{listing-dir}/imgHandler.ashx?ref={id}&order={N}&serial=1` (no image extension, not in raw HTML). The generic `<img>`/markdown extractors miss them, so Firecrawl returned 2 HTML links → validator rejected them as `text/html`. Confirmed live: the handler URL returns `200 image/jpeg` with NO referer/UA; this land listing has exactly 1 photo (order=1 image, order=2 = 404).
- **Altamira price** — page is server-rendered but the visible "€850,000" is JS-formatted; the value is in a hidden input `name="precio" value="850000" id="formOportunidad_precio"`. The existing regex looked for `id="precio"` (the id is actually `formOportunidad_precio`) so it never matched.

## Patch (1 file: services/portal-scraper.ts)
- Altamira price regex now matches `name="precio" value="…"` (+ flexible id/value-order fallbacks).
- New `fetchRemuGallery(url)`: derives the listing id + dir, probes `imgHandler.ashx` orders 1..N (cheap HEAD, batched, stop at first non-image, cap 40), and sets `result.imageUrls` to the real handler URLs for REMU. These validate (200 image/jpeg) and upload to Zyprus with no headers.
