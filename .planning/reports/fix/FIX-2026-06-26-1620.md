# Fix Report - 2026-06-26 16:20

**Symptom:** On approve and credit-note, Marios receives only a text/description, not the actual PDF.
**Mode:** general (1 file)
**Outcome:** fixed (most-likely cause; user to verify on live retest)

## Evidence gathered
- The approved invoice + credit-note PDFs DO generate and ARE stored in the invoice bucket (`invoices/generated/… Invoice 11445.pdf`, `… Credit Note 10106-10109.pdf`). So PDF generation/storage is NOT the problem.
- Approve flow: `sendDocumentToAccountingGroup` (group) → `notifyMariosApprovedAction` (Marios) fire two WhatsApp document sends back-to-back. Credit-note: `notifyGroupOfCreditNote` → `notifyMariosOverWhatsApp` — same pattern.
- `lib/whatsapp/client.ts sendDocument` retries the UPLOAD but not the SEND. Matches the known un-actioned audit finding "Vercel WhatsApp-client 429 retry".

## Root Cause
`lib/invoices/actions/documents.ts` `notifyMariosOverWhatsApp` — the Marios document send is the SECOND back-to-back WhatsApp document send (right after the accounting-group send). WaSender rate-limits (429) the second send; with no retry it failed, so Marios's copy was dropped. For credit notes (non-blank caption) it then fell through to a misleading "…PDF attached." TEXT with no actual PDF; for approvals (blank caption) it silently failed.

## Patch
- `lib/invoices/actions/documents.ts` (1 file): `notifyMariosOverWhatsApp` now retries the document send up to 2× with a 1.5s/3s backoff before any text fallback, so the back-to-back send to Marios reliably lands.
