# Fix Report - 2026-06-23 10:45

**Symptom:** (1) WhatsApp invoice with "included VAT" printed "V.A.T 16%" and Subtotal €35,000 (gross). (2) After approval the accounting group got the invoice but Marios did not get his copy with the right official number — regression, worked the day before.
**Mode:** general
**Outcome:** fixed

## Feedback Loop
- Initial: inspect PDF totals render + WaSender send path; `npx tsc --noEmit`.
- Final: `npx tsc --noEmit` PASS (exit 0); both root causes corrected in code.

## Root Cause
- Bug 1 — `lib/invoices/pdf.ts:239` — VAT rate was back-computed `Math.round(vatAmount / document.amount * 100)`. For `included-vat`, `document.amount` holds the GROSS (€35,000 = total), so `5588.24/35000 = 16%`, and the Subtotal line printed the gross instead of the net. The VAT *amount* itself was already correct 19%.
- Bug 2 — `lib/whatsapp/client.ts:267` — the document SEND step had no retry, while the UPLOAD step did (`uploadFileWithRetry`, maxRetries=2). Approval fires two document sends back-to-back (accounting group, then Marios); WaSender rate-limits the trailing send, so Marios's copy was dropped with no retry while the group's earlier send succeeded. Load-dependent → "worked yesterday".

## Patch
- Files changed: `lib/invoices/pdf.ts`, `lib/whatsapp/client.ts`
- Bug 1: rate is now fixed 19% (0 for `no-vat`) per Cyprus VAT — never back-computed; Subtotal derived as `|total| − |vatAmount|` so Subtotal + VAT = Total for every vatMode.
- Bug 2: send step wrapped in retry-with-exponential-backoff (2 retries), mirroring the upload retry, so every recipient (group AND Marios) reliably gets their copy under rate limiting. Text fallback preserved.
