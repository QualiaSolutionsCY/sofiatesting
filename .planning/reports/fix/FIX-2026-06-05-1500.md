# Fix Report - 2026-06-05 15:00

**Symptom:** "Send to Marios for review" isn't actually sending Marios a WhatsApp message.
**Mode:** quick
**Outcome:** fixed

## Feedback Loop

- Final passing check: `npx tsc --noEmit` → exit 0; send path now invokes the real WaSenderAPI client (`getWhatsAppClient().sendDocument`). Live WhatsApp delivery is the final confirmation (requires deploy + a real send).

## Root Cause
`lib/invoices/integrations/manual-provider.ts:13-24` — the "send to Marios" delivery is handled by `deliverWithManualProvider`, which only returns `deliveryStatus: "manual-copy-ready", copied: false` and makes **zero** WhatsApp API calls. The redesign dashboard nonetheless shows the toast "Draft sent to Marios via WhatsApp" (`components/invoices/redesign/App.tsx:191`), so the UI claims a send that never happens. A working sender already exists in this app (`lib/whatsapp/client.ts`, used by `app/api/documents/send/route.ts:247`) but was never wired into the invoice flow.

## Patch
- Files changed: `lib/invoices/actions/documents.ts` (1 file)
- Change summary: `sendToMariosAction` now calls a new best-effort `notifyMariosOverWhatsApp(document)` helper after queueing the delivery record. The helper resolves Marios's MSISDN from `INVOICE_AUTHORIZED_AGENTS` (`35799921560`), renders the invoice PDF via `buildDocumentPdfBytes`, and sends it to Marios with a review-request caption through the existing `getWhatsAppClient()` (`sendDocument`, falling back to `sendMessage` text on document-send failure). Send errors are logged but never break the status transition. The manual queue/delivery record is preserved so the delivery-history and retry UI keep working.
