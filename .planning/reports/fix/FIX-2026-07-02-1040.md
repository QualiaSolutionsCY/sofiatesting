# Fix Report - 2026-07-02 10:40

**Symptom:** The number of invoices under the "Paid" filter doesn't match the number of receipts. Expectation: paying an invoice issues a receipt, so the two counts should be 1:1.
**Mode:** quick
**Outcome:** fixed

## Feedback Loop
- Initial failing check: inspection — the "Paid" filter (`ListPane.tsx:17`) resolves to `STAGES.SENT_TO_ACCOUNTING.id`, and the fall-through predicate (`ListPane.tsx:83-87`) matches every invoice whose `stage === "sent-to-accounting"`, regardless of payment.
- Final passing check: `npx tsc --noEmit` (exit 0) + `pnpm exec tsx --test lib/invoices/*.test.ts` (33/33 pass); "Paid" filter now matches only invoices with `paidOn`/`receiptNo`, which are set 1:1 with a receipt.

## Root Cause
`components/invoices/redesign/ledger/ListPane.tsx:83-87` (+ chip at `:167`) — the "Paid" tab filtered by the `sent-to-accounting` lifecycle stage, but an invoice enters that stage on **approval** (`workflow-actions.ts:43-48` `markApproved` → `transitionDocumentStatus(document, "sent-to-accounting")`), before any payment. Payment (`workflow-actions.ts:117-146` `markPaidWithReceipt`) *also* lands at `sent-to-accounting` but additionally sets `paidAt`/`receiptNumber` and mints the receipt atomically. So the "Paid" tab counted approved-but-unpaid invoices as paid, inflating it above the Receipts count (`kind === "receipt"`, which only exists for genuinely-paid invoices). The paid⇒receipt coupling itself was never broken.

## Patch
- Files changed: `components/invoices/redesign/ledger/ListPane.tsx`
- Change summary:
  1. Added a dedicated "Paid" filter branch: an invoice shows under "Paid" only when actually paid (`doc.paidOn || doc.receiptNo`), not merely sent-to-accounting. This makes the Paid count equal the Receipts count 1:1.
  2. Fixed the row pay-chip (`:167`): dropped the `|| doc.stage === SENT_TO_ACCOUNTING.id` fallback so it reflects real payment; approved-but-unpaid invoices (incl. those sent to accounting) now read "Approved" instead of a false "Paid".

If a mismatch persists after deploy, it's one of these, not the stage conflation.
