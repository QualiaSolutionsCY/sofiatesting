# Fix Report - 2026-06-28 15:40

**Symptom:** (1) After approve/cancel the list + detail don't update ‚Äî the operator must hit Refresh. (2) New-invoice composer's unit-price shows a leading "0" / fragile decimals.
**Mode:** quick (frontend) ‚Äî found during the live panel e2e.
**Outcome:** fixed

## Root Cause
1. `components/invoices/redesign/App.tsx` ‚Äî approve handler reconciled the UI at the END of the transition, AFTER the best-effort `notifyAccountingGroupOfInvoiceAction` / `notifyMariosApprovedAction`; a throw in either (e.g. the Wasender token hiccup) hit the `catch` and SKIPPED `reconcile`, leaving a stale "DRAFT". The cancel handler reconciled right after `cancelWithCreditNoteAction` but had NO catch, so a throw from its internal group-send skipped the reconcile too. Either way the mutation persisted server-side but the UI stayed stale until a manual Refresh.
2. `components/invoices/redesign/modals/Composer.tsx` ‚Äî unit-price `<input type="number" value={l.unitPrice}>` bound to an initial numeric `0` (leading-0 to delete; fragile decimals on a number input).

## Patch (2 files)
- `App.tsx`: move `reconcile(result.documents, current.id)` + `setFilters(stage:"all")` to run IMMEDIATELY after `approveDocumentAction`, before the notifications; add `refetchDocuments()` in the approve `catch`. Wrap the cancel‚Üícredit-note path in try/catch with `refetchDocuments()` on error. ‚Üí the UI now updates the instant a mutation persists, regardless of delivery side-effects.
- `Composer.tsx`: unit-price ‚Üí `type="text" inputMode="decimal"`, `value={l.unitPrice || ""}` (empty instead of a stuck 0), decimal-guarded onChange. Totals/submit already coerce via `Number()`.
