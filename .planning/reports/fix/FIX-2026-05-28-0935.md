# Fix Report — sophia-bot portal-scraper missing fields

**Symptom:** Sofia regularly returns "Could not extract: Price, Covered area, Location, Bathrooms" when agents send bank-portal URLs (Altia, Altamira, REMU, Gordian). Extraction quality is too low to be useful.

**Mode:** general

**Outcome:** fixed (static root-cause patch; live-URL verification still recommended)

## Root Cause

Four overlapping defects in the Firecrawl extraction pipeline, all in `supabase/functions/sophia-bot/services/portal-scraper.ts`:

1. **Vague per-portal prompts** (`portal-scraper.ts:170-191`, old) — each prompt was 2-3 generic sentences ("look for bedrooms, bathrooms, area"). They never named the actual labels each portal uses ("Covered Area" vs "Built-up Area" vs "Internal Area" vs "Habitable Area"), never instructed price cleanup ("strip € and commas"), and never told Firecrawl that REMU/Gordian are JS-rendered SPAs whose details only appear post-render.
2. **Loose schema descriptions** (`portal-scraper.ts:99-168`, old) — schema field descriptions were one-liners ("Number of bathrooms"). LLMs guided by Firecrawl's structured-extract prompt use these descriptions; without label hints + cleaning rules, the model gave up easily on values formatted as "€350,000" or "125 m²".
3. **No SPA-render wait** (`portal-scraper.ts:268-275`, old) — `/v1/scrape` body had no `waitFor` or `onlyMainContent`. REMU and Gordian are JS SPAs; default scrape captures the initial empty shell, so the price/specs panel is just absent from what the extractor sees.
4. **Strict numeric merge** (`portal-scraper.ts:318-375`, old) — `mergeFirecrawlData` discarded any field whose runtime type wasn't `number`. When the LLM returned `"350000"` or `"€ 350,000"` (which it often does despite the schema requesting `number`), the value was silently dropped.

The four together explain the symptom: even when Firecrawl returns *something*, fields commonly come back as strings, missing, or unparseable — and the code happily drops them and tells the agent "couldn't extract."

## Patch

- **Files changed:** `supabase/functions/sophia-bot/services/portal-scraper.ts` (1 file).
- **Change summary:**
  - Rewrote each of the 4 portal prompts with concrete label hints (Altia: "Surface"/"Built area"/"Habitaciones"; Altamira: "Covered Area"/"Plot Area" + breadcrumb path; REMU: SPA-wait note + "Built-up Area"/"Land Area" + POA handling; Gordian: SPA-wait + URL-slug cross-check rules), explicit price-cleanup instructions, and per-portal location-extraction guidance.
  - Strengthened every schema field `description` to name the on-page labels, units, and cleanup rules — so the LLM is guided even when the prompt is summarised.
  - Loosened `propertyType` from a closed enum to a free-form lowercase string (bank portals carry plenty of commercial inventory — office/shop/warehouse/industrial/building — that the old enum silently dropped).
  - Added `onlyMainContent: true` and `waitFor: 2500` to the `/v1/scrape` body so JS-rendered portals (REMU, Gordian, Altia) finish hydrating before extraction runs.
  - Added a `coerceNumber()` helper that accepts numbers AND strings like `"€ 350,000"`, `"350.000"`, `"125 m²"`, `"3 beds"`. `mergeFirecrawlData` now routes `price`, `bedrooms`, `bathrooms`, `coveredArea`, `plotSize` through it, with rounding for integer counts and positive-value guards.
