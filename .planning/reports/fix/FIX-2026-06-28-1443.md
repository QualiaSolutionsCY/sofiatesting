**Symptom:** In the Monthly-run "drafts ready for your review" modal, the amount field shows a `0` that can't be removed — clearing it snaps straight back to `0`, so you can't type a fresh amount. Reported as "the zero I can't remove… it's always there when I edit a number."

**Outcome:** fixed

- Final: `npx tsc --noEmit` clean for the file; dev server (localhost:3000) hot-recompiled with no errors.

## Root Cause
`components/invoices/redesign/overlays/MonthlyRun.tsx:376-383` — a controlled `<input type="number" value={r.sub}>` with `onChange={(e) => editRow(r.id, { sub: Number(e.target.value) || 0 })}`. `r.sub` is the numeric net amount that drives `net`/`total`/the grand total (lines 157-162, 276), so it must stay a number. But clearing the field gives `e.target.value === ""` → `Number("")` is `0` → `0 || 0` is `0`, and `value={r.sub}` immediately re-renders the field as `"0"`. A numeric controlled input also can't hold an in-progress decimal like `595.`. Net effect: the `0` is un-removable and decimals are fragile.

## Patch
- Files changed: `components/invoices/redesign/overlays/MonthlyRun.tsx` (1 file)
- Change summary: Added a per-row raw-text buffer `amountText: Record<string, string>`. The amount input is now `type="text" inputMode="decimal"`, displays `amountText[r.id] ?? (r.sub === 0 ? "" : String(r.sub))` so a cleared/zero field shows the placeholder instead of a stuck `0`, accepts an empty string and a single in-progress decimal (regex-guarded `^\d*\.?\d*$`, letters rejected), and keeps the numeric model in sync via `editRow(... sub: raw === "" ? 0 : Number(raw) || 0)`. On blur the buffer is dropped so the field re-renders the normalized number. Totals/`r.sub` semantics unchanged.
