**Theme.** The dominant failure class is *misleading success on money-handling paths*: nearly every send path either routes through a no-op manual-provider stub or discards the real send result, then reports success to the operator/agent. This is the exact "admin/Sophia says it sent but nothing arrived" complaint Marios raised. The single CRITICAL is a tax/audit data-integrity defect (duplicate legal numbers under concurrency).

---

## CRITICAL

- **Official invoice/receipt/credit-note numbers have NO DB uniqueness — concurrent approvals produce duplicate legal numbers (read-modify-write race)** — `lib/invoices/actions/documents.ts:198-208 + lib/invoices/numbering.ts:53-65` — The official number is derived purely in app code (`Math.max(...numbers) + 1`) from a stale in-app list with no transaction, advisory lock, FOR UPDATE, or RPC, and the live DB has only `invoice_documents_external_id_key UNIQUE (external_id)` + the pkey — no UNIQUE on `(kind, official_number)`. Four independent concurrent writers exist (admin server actions, the Sophia intent bridge, the WhatsApp ✓-approval reply at `lib/invoices/supabase/webhook-repository.ts:86-89`, and the create flow); two racing approvals/issuances both read the same max and upsert the same number, both succeed silently — two legally-distinct tax documents get the SAME official number. Confirmed against live project `vceeheaxcrhmpqueudqx`. — Fix: Add a partial unique index `CREATE UNIQUE INDEX ON invoice_documents (kind, official_number) WHERE official_number IS NOT NULL AND deleted_at IS NULL;`, assign the number inside a Postgres transaction/sequence (SELECT max … FOR UPDATE or a per-kind sequence table) instead of computing from a stale in-app list, and retry on unique-violation.

## HIGH

- **"Resend both" (accounting CC) toast claims WhatsApp+Email delivery, but the action only queues a manual provider that sends nothing** — `components/invoices/redesign/App.tsx:566-571` — The toast `"Accounting CC resent (WhatsApp + Email)."` is unconditional; `forwardAccountingAction → queueAccountingHandoff → deliverWithManualProvider` returns `manual-copy-ready` / `copied:false` and sends nothing on any channel (no email is even queued on this path). The accounting group receives nothing while the UI reports success. — Fix: Route through the real `sendDocumentToAccountingGroup`, gate the toast on its boolean return, and drop the false "Email" claim.
- **Approve-from-draft toast claims "sent to Marios and the accounting group" regardless of whether either WhatsApp send succeeded** — `components/invoices/redesign/App.tsx:270-278` — `notifyAccountingGroupOfInvoiceAction` returns a boolean (false on missing MSISDN / unconfigured WhatsApp, `documents.ts:504-513`) and `notifyMariosApprovedAction` swallows all failures; both return values are discarded and the success toast fires unconditionally. On a Vercel deploy missing `WASENDER_API_KEY` / `INVOICE_ACCOUNTING_GROUP_MSISDN`, both sends are no-ops while the operator is told it was delivered. — Fix: Capture the accounting-group boolean and add a success signal to the Marios send; word the toast from both (e.g. "delivery to Marios/accounting could not be confirmed").
- **Auto-approve-on-create toast asserts "sent to the accounting group and Marios" even though sends are best-effort no-ops/failures** — `components/invoices/redesign/App.tsx:627-636` — All three sends (`notifyAccountingGroupOfInvoiceAction`, `notifyMariosApprovedAction`, `autoEmailApprovedInvoiceAction`) are awaited with every return value discarded, then the toast fires unconditionally. `autoEmailApprovedInvoiceAction` returns false when `RESEND_API_KEY`/`INVOICE_MARIOS_EMAIL` are unset (`documents.ts:625-630` — `RESEND_API_KEY` is missing on Vercel), and the WhatsApp sends fail silently. On every create the operator is told it was delivered when zero channels may have sent. — Fix: Collect the three booleans and only claim each channel that returned true.
- **`sendInvoiceEmailAction` throws on missing `RESEND_API_KEY`; client "Send email" shows a raw config error to the operator** — `lib/invoices/actions/documents.ts:344-345` — `if (!apiKey) throw new Error("Email isn't configured (missing RESEND_API_KEY).")` is the first statement; `RESEND_API_KEY` is set on the edge fn but not on Vercel, so the `client-send-all` handler (`App.tsx:509`) throws and surfaces the raw message verbatim. Emailing an invoice/receipt/credit note to a client from the admin panel is completely broken; the button (`DetailPane.tsx:610-614`) is gated only on `!hasNumber`, never on config. — Fix: Set `RESEND_API_KEY` on Vercel (env pull/add) and gate the "Send email" button on a server-reported config flag so it is disabled rather than failing on click.
- **`sendDocumentByUrl` failure is ignored — `documentSent` set true unconditionally, so the agent gets NEITHER the PDF NOR the text reply (silent total failure)** — `supabase/functions/sophia-bot/tools/handlers/invoice.ts:112-134` — `sendDocumentByUrl` (`utils/wasend.ts:578-624`) returns a Response on every outcome (503 on breaker-open `:585-588`, 500 on fetch error `:622`) and never throws; the handler ignores it and sets `documentSent = true`, then `webhook.ts:377-378` (`if (aiResult.documentSent) return tokenCount;`) suppresses the text reply. On any WaSend failure the agent who created/approved/receipted gets neither the PDF nor the confirmation text — the "Non-fatal — the text reply still goes out" comment is false. — Fix: `const res = await sendDocumentByUrl(...); documentSent = res.ok;` — leave `documentSent=false` on a non-ok response so the webhook falls through to the text reply; log the non-ok status.
- **Sophia `sendDocumentByUrl` returned HTTP failure is never checked — `documentSent` set true even on 500/503, and a phantom "last document" is stored** — `supabase/functions/sophia-bot/tools/handlers/invoice.ts:111-134` — Same root cause as above, additionally: `saveLastDocument` runs without checking `res.ok` (`invoice.ts:125-130`), registering an undelivered PDF as the user's "last document" that a later "email it to me" would attach. — Fix: Inspect the Response; only set `documentSent=true` and call `saveLastDocument` when `res.ok`, otherwise log and leave `documentSent=false`.
- **Approve intent claims "sent a copy to Marios" but the Marios-send result is discarded — it only checks the group boolean** — `lib/invoices/sophia/intent-handlers.ts:216-224` — The reply branches solely on `sentToGroup`; `notifyMariosApprovedAction` is awaited with its result thrown away, and `notifyMariosOverWhatsApp` is `Promise<void>` that swallows every failure (`documents.ts:191-195`). When the group send succeeds but the Marios send fails, Sophia tells the approver the copy reached Marios — it did not, and nobody is alerted. Reachable via `app/api/sophia/intent/route.ts:61-62` → WhatsApp reply. — Fix: Make `notifyMariosOverWhatsApp`/`notifyMariosApprovedAction` return a boolean, capture it, and word the reply from BOTH flags.
- **`request_correction` and `resend` intents send nothing to the team but Sophia reports success ("Marked for correction & resend" / "Resent … PDF attached")** — `lib/invoices/sophia/intent-handlers.ts:305-310` — Both route through `correctResendAction → queueCorrectedResend → deliverWithManualProvider` (no-op stub), and the queue row is marked `sent`, yet no WhatsApp/email goes out. The admin "Correct & resend" uses the REAL sender `resendCorrectedInvoiceAction` (`documents.ts:584-617`); Sophia's intents take the dead path. For `resend`, the PDF attaches only to the requester's own WhatsApp — never re-sent to Marios or the group despite the word "Resent". — Fix: Wire these intents to `sendDocumentToAccountingGroup` + `notifyMariosOverWhatsApp` (as `resendCorrectedInvoiceAction` already does), or change the replies to state the truth.
- **Receipt PDF (`mark_paid` / `issue_receipt`) is delivered ONLY by the edge handler with `notifyMarios:false` — combined with the `documentSent` bug, a failed send loses the receipt silently** — `lib/invoices/sophia/intent-handlers.ts:313-329` — With `notifyMarios:false` the server action performs zero delivery (`documents.ts:404-405`), so the requester's `sendDocumentByUrl` is the only delivery; the `documentSent`-true-on-failure bug suppresses the text, so on a WaSend failure the requester gets no PDF and no text and Marios was never sent a copy. The receipt is issued + numbered in the DB but delivered to no one, with no error. — Fix: Fix the `documentSent` gating and/or keep `notifyMarios:true` (or add an explicit group/Marios send) so the receipt has a reliable delivery path.
- **Approve reply claims "sent a copy to Marios" based only on the GROUP send — a failed Marios delivery is reported as success** — `lib/invoices/sophia/intent-handlers.ts:216-224` — (Distinct instance of the approve-reply gap, asymmetry confirmed: `sendDocumentToAccountingGroup` returns `Promise<boolean>` while the Marios path returns `void`.) On a failed Marios WhatsApp send, accounting records the invoice as delivered to Marios when it was not. — Fix: Thread a boolean from the Marios send and word the reply from both flags, naming exactly who was reached.
- **Approve reply claims "sent a copy to Marios" even when the Marios WhatsApp send fully failed (misleading financial confirmation)** — `lib/invoices/sophia/intent-handlers.ts:216-224 + lib/invoices/actions/documents.ts:117-122,191-195` — Same defect from the action layer: `notifyMariosOverWhatsApp` returns void and swallows errors (`catch (error) { sendLogger.error(...) }`), so no success signal can reach the reply. — Fix: Make the Marios send return a boolean, propagate through `notifyMariosApprovedAction`, gate the reply.
- **Blank-caption document send to Marios has NO working fallback — the text fallback re-sends an empty string that WaSender rejects** — `lib/invoices/actions/documents.ts:184-190` — For receipts (caption `""`, `documents.ts:160-161`) and blank-rule invoices, when the PDF send fails the fallback calls `sendMessage({ text: caption })` with `text:""`; WaSender rejects empty/whitespace text (the documented reason `client.ts:277` OMITS the field for documents), so the fallback also fails. Marios silently never receives the receipt/invoice while the flow reports success. Same flaw in `sendDocumentToAccountingGroup` (`documents.ts:521-524`). Reachable via the admin-panel receipt path (`markPaidAndIssueReceiptAction`, `notifyMarios` defaults true). — Fix: In the fallback substitute a minimal non-empty text when caption is blank, or surface the failure to the caller. (Note: the `notifyGroupOfCreditNote` path at `documents.ts:486-488` is NOT affected — its caption is always non-empty.)
- **Manual-provider delivery queue is a no-op that sends nothing, yet is recorded as `sent` / `manual-copy-ready` and rendered to the operator as a completed delivery** — `lib/invoices/integrations/manual-provider.ts:13-24` — `deliverWithManualProvider` returns `queueStatus:"sent"`, `deliveryStatus:"manual-copy-ready"`, `copied:false` with no network call; `integration-repository.ts:101-108` writes status `sent` + provider id + `sent_at`, `:112-121` inserts a delivery event, and `:323-330` inserts a legacy `message_events` row hardcoded `status:"sent"`. The admin Delivery panel (`DocumentDetail.tsx:476-487`, timeline `:528-534`) renders it with channel, provider id, and attempts — indistinguishable from a real send, with no "copy manually" indication. It is the ONLY delivery backend wired to live operator actions. — Fix: Wire it to the real WhatsApp client/Resend, or relabel its output (`needs-manual-copy` / `pending`) and render it as a TODO, not as "sent".
- **Admin-panel "Forward to accounting" sends nothing to WhatsApp — it only enqueues the no-op manual provider, diverging from the Sophia approve path which uses the real sender** — `lib/invoices/actions/documents.ts:300-307` — `forwardAccountingAction` calls only `queueAccountingHandoff` (no-op), never `sendDocumentToAccountingGroup`. The document flips to `sent-to-accounting` and a delivery record appears, but the group receives nothing. Live trigger is "Resend both" (`DetailPane.tsx:645` → `App.tsx:568`) with the hardcoded success toast. — Fix: Call `sendDocumentToAccountingGroup(updated, caption)` for the real send (same caption rule as `notifyAccountingGroupOfInvoiceAction`), keep `queueAccountingHandoff` only for the audit record, and gate the status transition/toast on the real result.
- **Approval audit trail is destroyed on every read/save round-trip — `invoice_approvals` never written, timeline collapsed to one synthetic entry** — `lib/invoices/supabase/document-mappers.ts:205-211 + lib/invoices/supabase/document-repository.ts:202-241` — `fromDocumentRow` replaces real history with a single fabricated entry (`"Loaded from Supabase document row" / "Sophia"`); `writeRelatedRows` never calls `toApprovalRows` (referenced only by dead `seed-documents.ts`). Live DB confirms `invoice_approvals` has 0 rows while `invoice_documents` has 59 and `invoice_document_revisions` has 225 — saves run constantly while the audit table stays empty, and revision snapshots persist the already-collapsed timeline. For a financial system there is no durable record of who approved each invoice or when. — Fix: Insert `toApprovalRows` (or append-only events) in `writeRelatedRows`, and rebuild `approvalTimeline` in `fromDocumentRow` from `invoice_approvals`/`invoice_document_revisions` instead of fabricating one entry.

## MEDIUM

- **`deliverWithManualProvider` is a no-op stub: every queued WhatsApp/email delivery is recorded as "sent" without sending** — `lib/invoices/integrations/manual-provider.ts:12-24` — Every queue route writes a row with `queueStatus:"sent"` but performs zero delivery (`copied:false`). The operator-visible delivery card honestly shows `deliveryStatus` ("manual copy ready"), but the persisted `queueStatus:"sent"` and the legacy `message_events status:"sent"` misrepresent the audit trail; the genuinely misleading callers are `forwardAccountingAction` and `correctResendAction`, which route ONLY through the stub yet show success toasts. — Fix: Wire to the real senders, or change the recorded status to a non-"sent" state so the audit trail does not represent these as delivered.
- **Approve fires group + Marios PDF sends back-to-back with no inter-send delay, risking WaSender rate-limit (429) drops on the second send** — `lib/invoices/sophia/intent-handlers.ts:216-217` — Two uploads+sends fire immediately in sequence (and `cancelWithCreditNoteAction` does the same, `documents.ts:441,443`); the Vercel `WhatsAppClient.sendDocument` has no 429/backoff path (unlike edge `wasend.ts`), so the second send can hit 429 and is dropped with no retry and no user-visible error (result discarded). — Fix: Add a short inter-send delay and 429-aware retry to the Vercel client's `sendDocument`, and gate the reply on each result.
- **Receipt/credit-note draft sequence uses total document COUNT instead of per-kind max — meaningless, collision-prone draft numbers** — `lib/invoices/actions/documents.ts:393,421 + lib/invoices/numbering.ts:4-8` — `current.documents.length + 1` (all kinds, currently 58) feeds `createDraftNumber`, unrelated to the receipt/credit-note sequence space; deleting a document shrinks the count so the next draft can reuse a prior sequence (no DB unique on `draft_number`). — Fix: Replace both with `getNextDraftSequence(current.documents, 'receipt')` / `'credit-note'`.
- **`getNextOfficialNumber` strips ALL non-digits before `Math.max` — a year-prefixed official number poisons the sequence** — `lib/invoices/numbering.ts:53-65` — `Number(number.replace(/\D/g, ""))` turns `'2026-11424'` into `202611424`; since `official_number` is client-supplied free text, the first prefixed/separated value jumps the whole per-kind sequence to a 9-digit value — one-way corruption (latent: current data is pure-digit). — Fix: Extract only the trailing numeric group (reuse `extractSequence`), and validate official numbers on input.
- **Upsert always writes `deleted_at` from the in-memory copy — a save can silently resurrect a soft-deleted invoice** — `lib/invoices/supabase/document-mappers.ts:157 + lib/invoices/supabase/document-repository.ts:66-72` — `toDocumentRow` always emits `deleted_at: document.deletedAt ?? null` and `saveInvoiceDocument` upserts the full row on `external_id` with no guard; any path that regenerates a document without its `deletedAt` (retry, PDF regenerate, stale client action) clears `deleted_at` and brings the invoice back into the live ledger, bypassing `restoreInvoiceDocument`. — Fix: Omit `deleted_at` from the normal save path (manage it only via the dedicated delete/restore updates), or guard the write so a soft-deleted row can only be revived through restore.
- **`issue_credit_note` relabels the user's group message as a cancellation "reason", conflicting with what the prompt promises** — `lib/invoices/sophia/intent-handlers.ts:339-340` — `params.groupMessage` is fed into `cancelWithCreditNoteAction` as the `reason`, which is printed on the PDF as "Invoice cancelled. Reason: …", appended to the description, and woven into the group caption — so "FYI per Marios" becomes "Invoice cancelled. Reason: FYI per Marios" on the legal document. — Fix: Pass `params.correctionReason` only as the cancellation reason, and pass `groupMessage` through to `notifyGroupOfCreditNote` as the caption.
- **`findDoc` client-name match is a loose substring with no disambiguation — can act on the WRONG invoice** — `lib/invoices/sophia/intent-handlers.ts:86-91` — For approve / edit / mark_paid / issue_credit_note / resend with no `documentId`, the doc is resolved by the first `clientName` containing the substring (ordered official_number desc); with multiple invoices per client (common for recurring) Sophia can approve, mark-paid, or issue an irreversible credit note against the wrong financial document with no warning. — Fix: On multiple matches with no `documentId`, return `ok:false` with a disambiguation reply listing candidate numbers.
- **`manageInvoice` tool exposes `recurrence` but not `recurrenceDay`, so monthly/yearly invoices can't get a billing day from Sophia** — `supabase/functions/sophia-bot/tools/definitions.ts:788-792` — The tool definition has no `recurrenceDay` property even though the zod schema, edge handler, `IntentParams`, and `create_draft` all support it; the model never sees the field, so every recurring invoice Sophia creates has `recurrenceDay=undefined`. — Fix: Add a `recurrenceDay` integer (1-31) property to the tool definition.
- **"Mark notifications muted" and "Schedule" are pure UI lies — they show a success toast and do nothing** — `components/invoices/redesign/App.tsx:497-498` — `whatsapp-marios-mute` and the scheduling action (`App.tsx:563-564`) report a state change but persist nothing and call no action. — Fix: Implement (persist a mute flag / scheduled-send row) or remove the buttons; if staged, change the copy to say it is not yet active.
- **"Duplicate as new draft" creates a client-only document that is never persisted and silently vanishes on refresh** — `components/invoices/redesign/App.tsx:363-387` — Duplicate only mutates local React state with a random client id; it never calls `createDocumentAction`, so the focus/visibility refetch and Refresh replace `docs` with the server set and the draft disappears; editing it would call `updateDocumentAction` with an id the server doesn't have. — Fix: Call `createDocumentAction` and reconcile the server result.
- **Async server-action handlers have no loading state; rapid double-clicks fire duplicate mutations** — `components/invoices/redesign/App.tsx:129` — `isPending` from `useTransition` is discarded, so no button is disabled during an await; double-clicking "Mark as paid → issue receipt" or "Approve & send" issues two receipts/approvals/group posts. — Fix: Destructure `isPending`, thread it to `DetailPane`/`DeliveryPlan`, disable buttons and show a pending indicator while a transition is active.
- **Composer pushes new clients into the shared in-memory `CLIENTS` array, which is wiped by every server reconcile** — `components/invoices/redesign/modals/Composer.tsx:164-176` — New clients are mutated into the module-level `CLIENTS` registry which `replaceClientRegistry` overwrites on every reconcile; the Composer assigns `c<n>_<rand>` ids while the adapter derives `client-<slug>` ids, so after the create reconciles `clientById` returns the "Unknown" fallback in any subsequent edit/preview until reload. — Fix: Resolve the client by name against the server-derived registry (or key to `clientIdFromName(name)`) instead of inventing an id.
- **DeliveryPlan shows a fabricated client email address while the actual send prompts for one — the displayed address is fiction** — `components/invoices/redesign/ledger/DetailPane.tsx:519-523` — The client card displays an invented `@example.com` address (and "Delivered"/"Ready" states derived from doc stage), but `client-send-all` ignores it and `window.prompt()`s for an address; the card misrepresents both the destination and the delivery state. — Fix: Show the real client email (or "No email on file") and derive channel state from actual delivery records; remove the placeholder.
- **"Deleted" view has no loading or error state and silently shows stale/empty results on failure** — `components/invoices/redesign/App.tsx:196-208` — The deleted load has no pending indicator and only `console.error`s on failure, leaving the generic "No invoices match." empty state; an operator cannot tell whether there are no deleted docs or the fetch failed. — Fix: Track a loading flag and render a loading row; on catch set an error state with a retry affordance.
- **`sendToMariosAction` toast claims "bumped in CSC Review group" but the send targets Marios's personal number, and success is never checked** — `components/invoices/redesign/App.tsx:287-292` — The send goes to `marios.msisdn` (a personal number) and skips entirely if Marios is absent from `INVOICE_AUTHORIZED_AGENTS` or WhatsApp is unconfigured, while the toast unconditionally claims a group bump — wrong channel name and unverified delivery. — Fix: Return a success signal and gate the toast on it; correct the copy to direct WhatsApp to Marios.

## LOW

- **`create_draft` amount validation rejects amount 0 inconsistently with the schema, and accepts NaN-derived undefined paths** — `lib/invoices/sophia/intent-handlers.ts:118-119` — The guard checks only `typeof === "number"`, not `> 0`; the zod schema allows `min(0)`, so an amount-0 invoice (likely an LLM mistake) is created and queued to Marios as a €0.00 draft with no warning. — Fix: Reject non-positive amounts: `if (!params.client || typeof params.amount !== "number" || params.amount <= 0) return { ok:false, reply: "I need the client name and a positive amount to create a draft." };`

## Verdict

**FAIL** — 1 CRITICAL and 15 HIGH findings exist. The send pipeline (Delivery score 1, Edge-flow score 1) systematically reports success while delivering nothing, and the official-number sequence has no DB-level integrity guarantee.

Recommended next: `/qualia-fix` for the broken behaviors — start with the CRITICAL numbering race and the no-op delivery stub / unconditional success-toast cluster, since those produce the "says it sent but nothing arrived" reports Marios filed.
