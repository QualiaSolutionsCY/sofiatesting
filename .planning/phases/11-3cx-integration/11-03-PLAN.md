**Enhance client.ts with retry and session management:**

1. Wrap all fetch calls in `withRetry()` from sophia-bot utils:
   - Login: maxRetries=2, baseDelayMs=2000 (3CX may be slow to respond)
   - API calls: maxRetries=3, baseDelayMs=1000
2. Add session refresh logic to `makeAuthenticatedRequest()`:
   - If response is 401 or 403: call `this.login()` once, then retry the original request
   - If re-login also fails: throw `ThreeCXAuthError` with "Session expired and re-authentication failed"
   - Track re-login attempts to prevent infinite loops (max 1 re-login per request)
3. Add request timeout handling:
   - Set AbortController with 30-second timeout on all fetch calls
   - If timeout: log warning and throw with "3CX API request timed out after 30s"
4. Add response validation:
   - Check Content-Type is JSON before parsing
   - If HTML response (common for auth redirects): detect and throw "3CX returned login page - session likely expired"

**Enhance call-log-extractor.ts with graceful degradation:**

1. Wrap API response parsing in try/catch:
   - If response format doesn't match: log the actual response keys/structure, throw descriptive error
   - Include the 3CX version/endpoint that was used in the error message
2. Handle edge cases in call log data:
   - Missing callerNumber field: skip entry, log warning
   - Null or empty phone numbers: skip, log
   - Malformed date strings: use Date.parse fallback, log warning
3. Handle empty results gracefully:
   - No calls found: return valid CallAuditResult with empty externalCallers array (not an error)
   - API returns error but some data was already fetched: return partial results with errors array populated

**Enhance index.ts error reporting:**

1. Add structured error classification:
   - `AUTH_FAILED`: 3CX login failed (wrong credentials or endpoint)
   - `SESSION_EXPIRED`: Was authenticated but session died
   - `NETWORK_ERROR`: Cannot reach 3CX server
   - `PARSE_ERROR`: 3CX response format unexpected
   - `NO_DATA`: Successfully connected but no call data found
2. Each error response includes:
   - `success: false`
   - `error`: error class string
   - `message`: human-readable description
   - `timestamp`: ISO string
   - `retryable`: boolean (helps caller decide whether to retry)
3. Add execution timing:
   - Log start time, end time, and total duration
   - Include duration in response: `"executionMs": 1234`
4. Add a `?health` query parameter that checks:
   - Config is valid (env vars present)
   - 3CX is reachable (TCP connect test or HEAD request)
   - Returns health status without attempting full login
