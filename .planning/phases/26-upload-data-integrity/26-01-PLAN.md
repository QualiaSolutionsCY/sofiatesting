## Implementation Steps

### Step 1: Fix upload lock leak on all early-return paths (FR-1)

**File:** `supabase/functions/sophia-bot/tools/handlers/field-validation.ts`

**Problem:** Upload lock acquired at line 122 is only released on the duplicate-detection path (line 219). Six other early-return paths leak the lock, blocking the agent for the full TTL.

**Changes:**
1. After the lock is acquired (line 122-134), wrap all subsequent code that can return early in a try/finally pattern, OR add explicit `releaseUploadLock(propertyLockKey)` calls before each early return.
2. Specific paths that need lock release:
   - Line 236-265: `validateRequiredFields` failure
   - Line 325-328: Street address detected, area extracted from URL but asking agent
   - Line 330-343: Street address without Google Maps URL
   - Line 348-358: City-only location rejection
   - Line 403-405: Regional access denied
   - Line 421-423: Special case rejection
   - Line 425-427: Special case needs input
   - Line 430-435: Assignment input needed

**Approach:** Add a helper that wraps the release:
```typescript
const releaseLock = () => releaseUploadLock(propertyLockKey).catch(() => {});
```
Then call `await releaseLock()` before every early return after lock acquisition.

**Acceptance criteria:**
- [ ] Every `return` statement after line 134 that ISN'T the success path calls `releaseLock()`
- [ ] Successful path still releases lock (already does at property-listing.ts:521)

**Changes in ai-chat.ts:**

2. This preserves the AI's `bedrooms: 0` for studios during email uploads

**Acceptance criteria:**

### Step 4: Fix parsePreExtractedFields robustness (FR-4)

**File:** `supabase/functions/sophia-bot/services/ai-chat.ts`

**Problem:** The regex at line 1108 silently returns `{}` if the email format varies slightly. No warning logged.

**Changes:**
1. Add a warning log when the regex fails to match:
```typescript
if (!blockMatch) {
  logger.warn("[Email] parsePreExtractedFields: Failed to match PRE-EXTRACTED FIELDS block — AI will use its own extraction", {
    category: LogCategory.TOOL,
    messagePreview: userMessage.substring(0, 200),
  });
  return result;
}
```

2. Make the regex more resilient — allow for variations in delimiter:
```typescript
const blockMatch = userMessage.match(
  /PRE-EXTRACTED FIELDS[^\n]*\n([\s\S]*?)(?:\n\n(?:RULES:|MANDATORY|\[Subject:)|\n\[Subject:)/
);
```
This adds `MANDATORY` as an alternate delimiter and handles cases where `[Subject:` follows with only one newline.

**Acceptance criteria:**
- [ ] When regex fails, a WARN-level log is emitted with message preview
- [ ] Regex still matches the current format correctly
- [ ] Regex handles minor formatting variations (single vs double newline before Subject)

### Step 5: Fix poolType parsing corruption (FR-6)

**File:** `supabase/functions/sophia-bot/services/ai-chat.ts`

**Problem:** `poolType: "none" [warning text]` from email-parser's `formatExtractedFields` fails the `endsWith('"')` check, falling through to become a garbled string.

**Changes in `parsePreExtractedFields`:**
1. After extracting the raw value string, strip any text after a closing quote + space:
```typescript
// Handle values like: "none" [warning text] — strip everything after closing quote
if (strVal.startsWith('"')) {
  const closingQuoteIdx = strVal.indexOf('"', 1);
  if (closingQuoteIdx > 0) {
    value = strVal.slice(1, closingQuoteIdx);
  } else {
    value = strVal.slice(1); // no closing quote, take rest
  }
}
```
Replace the current `startsWith('"') && endsWith('"')` logic with this more robust approach.

**Acceptance criteria:**
- [ ] `poolType: "none" [warning]` correctly parses to `"none"`
- [ ] `poolType: "private"` still correctly parses to `"private"`
- [ ] `location: "Kato Paphos, Paphos"` still correctly parses to `"Kato Paphos, Paphos"`

### Step 6: Fix email Google Maps follow-up flow (FR-3)

**File:** `supabase/functions/sophia-bot/services/ai-chat.ts` and `supabase/functions/sophia-bot/handlers/email-webhook.ts`

**Problem:** When email upload blocks on missing Google Maps link (ai-chat.ts:782-794), the response is returned from `chat()` but:
1. The follow-up email is a NEW email (not "Re:"), so email-webhook.ts uses empty history
2. The agent's reply with the Google Maps link has no context of the original property

**Changes in ai-chat.ts:**
1. At line 782-794, instead of returning immediately, store the blocking response in the tool result and let the normal flow handle it. The response IS stored by email-webhook.ts:314-319 after `chat()` returns, so the real issue is step 2.

**Changes in email-webhook.ts:**
1. The problem is at line 241: `const isReply = /^re:/i.test(subject.trim())`. When the agent replies with a Google Maps link, the email-router may strip the "Re:" prefix or the agent might send a new email.
2. Fix: After loading history (or finding empty history for non-replies), check if the LAST stored message for this userId mentions "Google Maps link" or "pin location". If so, load the last 4 messages instead of 0, so the AI has context:
```typescript
if (!isReply && history.length === 0) {
  // Check if there's a recent "waiting for Google Maps" response
  try {
    const recentHistory = await getHistory(userId);
    const lastModel = recentHistory.filter(m => m.role === "model").pop();
    const lastText = lastModel?.parts?.[0]?.text || "";
    if (lastText.includes("Google Maps") || lastText.includes("pin location")) {
      history = recentHistory.slice(-4);
      logger.info("[Email] Non-reply but detected pending Google Maps request — loading recent history", {
        category: LogCategory.GENERAL,
      });
    }
  } catch { /* fall through to empty history */ }
}
```

**Acceptance criteria:**
- [ ] Email upload without Google Maps link returns "send me the Google Maps link" response
- [ ] Agent's follow-up email (new or reply) with the link has context to complete the upload
- [ ] Normal new emails still use empty history (no contamination)

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Lock release adds latency | `releaseUploadLock` is fire-and-forget with `.catch(() => {})` — negligible |
| Removing bedrooms default breaks existing flows | `validateRequiredFields` already has bedrooms in required list — it was just masked by Zod default |
| Email history loading for Google Maps follow-up could load wrong context | Only triggers when last AI response mentions "Google Maps" — very targeted |
| parsePreExtractedFields regex change could break current parsing | New regex is a superset — still matches current format, just also matches variations |
