**SEC-06 Fix: Prompt Update Size Limit**

In the PUT handler of `app/api/admin/prompts/[key]/route.ts`, add a size check after parsing the body but before any database operations.

After the existing content validation block (line ~108), add:

```typescript
// SEC-06: Reject oversized prompt content (50KB limit)
const MAX_PROMPT_SIZE = 50 * 1024; // 50KB
if (new TextEncoder().encode(content).byteLength > MAX_PROMPT_SIZE) {
  return NextResponse.json(
    { error: "Prompt content exceeds maximum size of 50KB" },
    { status: 413 }
  );
}
```

Place this immediately after the `if (!content || typeof content !== "string")` check so:
1. Content is confirmed to be a string
2. Size is checked before any DB queries
3. Uses byte length (not string length) for accurate size measurement

**SEC-06 Fix: Rollback Endpoint Size Limit**

The rollback endpoint creates a new version with the content from a target version. While the content comes from an existing DB record (not user input), we should still enforce the limit as a defense-in-depth measure on the request body itself.

Add a size check on the incoming request body. After `const body = await request.json();` (line ~42), add:

```typescript
// SEC-06: Reject oversized request payloads (50KB limit)
const MAX_PAYLOAD_SIZE = 50 * 1024; // 50KB
const bodySize = new TextEncoder().encode(JSON.stringify(body)).byteLength;
if (bodySize > MAX_PAYLOAD_SIZE) {
  return NextResponse.json(
    { error: "Request payload exceeds maximum size of 50KB" },
    { status: 413 }
  );
}
```

This protects against oversized request bodies even though rollback content comes from DB.
