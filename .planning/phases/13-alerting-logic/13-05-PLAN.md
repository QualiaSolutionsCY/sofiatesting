<objective>
Unify alert response tracking onto the `caller_alerts` table, eliminating the dual-table
architecture that causes Vasya's Telegram responses to update the wrong table.

Purpose: Fix the critical ALERT-04 blocker where follow-up reminders ignore resolved status
because the response tracker writes to `audit_alerts` while follow-up logic reads from
`caller_alerts`. After this fix, a single table (`caller_alerts`) owns all alert state.

Output: Updated response tracker, cleaned-up alert sender, migration for `chat_id` column.
</objective>

Key schemas:

**caller_alerts (Phase 10 — CORRECT table):**
- id UUID PK
- caller_phone TEXT
- audit_run_id UUID FK
- call_record_id UUID FK (nullable)
- call_time TIMESTAMPTZ (nullable)
- status TEXT (pending | alerted | follow_up_sent | resolved | ignored)
- alert_message_id TEXT (nullable) — stores Telegram message ID as string
- follow_up_message_id TEXT (nullable)
- resolution_type TEXT (nullable) — found_in_telegram | alternative_phone | not_client | manual_ignore
- resolution_note TEXT (nullable)
- alternative_phone TEXT (nullable)
- alerted_at, follow_up_at, resolved_at TIMESTAMPTZ (nullable)
- created_at, updated_at TIMESTAMPTZ
- UNIQUE(caller_phone, audit_run_id)
- **Missing: chat_id column** (needed for response lookup)

**audit_alerts (Phase 12 — DEPRECATED after this plan):**
- id UUID PK
- phone_number TEXT, call_date DATE, call_time TEXT
- alert_type TEXT, telegram_message_id BIGINT, chat_id BIGINT
- status TEXT (pending | resolved)
- response_text, responded_by_telegram_id, responded_at
- UNIQUE(phone_number, call_date, alert_type)

**Field mapping (audit_alerts → caller_alerts):**
- phone_number → caller_phone
- telegram_message_id (BIGINT) → alert_message_id (TEXT — must cast)
- status "resolved" → status "resolved" (same)
- status "pending" → status "alerted" (follow-up eligible, not "pending" which means pre-alert)
- response_text → resolution_note
- No direct mapping for responded_by_telegram_id — not in caller_alerts schema, skip
- No direct mapping for responded_at → use resolved_at for "resolved" status
