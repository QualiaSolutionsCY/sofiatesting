Create a SQL migration that sets up:

**1. Enable required extensions (idempotent):**
```sql
CREATE EXTENSION IF NOT EXISTS pg_cron;
```

**2. Execution log table:**
Create `cron_execution_log` table to track each cron invocation:
- `id` SERIAL PRIMARY KEY
- `job_name` TEXT NOT NULL (e.g., 'call-audit-daily')
- `scheduled_at` TIMESTAMPTZ NOT NULL DEFAULT NOW()
- `started_at` TIMESTAMPTZ
- `completed_at` TIMESTAMPTZ
- `status` TEXT NOT NULL DEFAULT 'scheduled' (scheduled, running, success, failed)
- `http_status` INTEGER (response code from Edge Function)
- `response_body` TEXT (truncated to 2000 chars)
- `error_message` TEXT
- `execution_ms` INTEGER

Add index on `(job_name, scheduled_at DESC)` for querying recent executions.
Enable RLS, add service_role policy (same pattern as call_audit_runs).

**3. Wrapper function for logged invocation:**
Create a PL/pgSQL function `invoke_call_audit()` that:
1. Inserts a row into `cron_execution_log` with status='running', started_at=NOW()
2. Calls `net.http_post()` to invoke the Edge Function:
   - URL: `https://vceeheaxcrhmpqueudqx.supabase.co/functions/v1/call-audit`
   - Headers: `{"Authorization": "Bearer " || current_setting('app.settings.service_role_key', true), "Content-Type": "application/json", "x-cron": "true"}`
   - Body: `'{}'`
3. Note: `net.http_post` is async (fire-and-forget), so the function just records that invocation was triggered. The Edge Function's own logging (call_audit_runs) handles result tracking.
4. Updates the log row with status='success' (meaning: HTTP request dispatched successfully)
5. On exception: updates status='failed' with error_message

IMPORTANT: Use `current_setting('app.settings.service_role_key', true)` for the auth header. If this doesn't work on Supabase, use a hardcoded placeholder `<SERVICE_ROLE_KEY>` with a comment that it must be replaced. The draft-cleanup pattern shows `<service_role_key>` as placeholder.

Actually, follow the EXACT same pattern as draft-cleanup (simplest, proven to work):
```sql
SELECT cron.schedule(
  'call-audit-daily',
  '0 17 * * 1-5',
  $$SELECT net.http_post(
    url := 'https://vceeheaxcrhmpqueudqx.supabase.co/functions/v1/call-audit',
    body := '{}'::jsonb,
    headers := '{"Authorization": "Bearer <SERVICE_ROLE_KEY>", "Content-Type": "application/json", "x-cron": "true"}'::jsonb
  )$$
);
```

Use the `timezone` parameter of `cron.schedule()`:
```sql
SELECT cron.schedule(
  'call-audit-daily',
  '0 17 * * 1-5',
  $$...$$,
  'Europe/Nicosia'
);
```

Wait -- `cron.schedule` in pg_cron has this signature: `cron.schedule(job_name, schedule, command)`. The timezone parameter was added in pg_cron 1.5+ as a 4th parameter. Supabase uses pg_cron 1.5+, so this should work. BUT if `cron.schedule` doesn't accept 4 args on this Supabase version, use the alternative:
```sql
UPDATE cron.job SET timezone = 'Europe/Nicosia' WHERE jobname = 'call-audit-daily';
```

Use the 3-argument form first, then UPDATE timezone separately (safest):
```sql
SELECT cron.schedule('call-audit-daily', '0 17 * * 1-5', $$...$$);
UPDATE cron.job SET timezone = 'Europe/Nicosia' WHERE jobname = 'call-audit-daily';
```

The cron expression `0 17 * * 1-5` means: minute 0, hour 17, any day of month, any month, Monday through Friday. With timezone set to Europe/Nicosia, PostgreSQL handles EET (UTC+2) and EEST (UTC+3) transitions automatically.

**4. Log cleanup job:**
Schedule a weekly cleanup to prevent unbounded growth:
```sql
SELECT cron.schedule(
  'cron-log-cleanup',
  '0 3 * * 0',
  $$DELETE FROM cron_execution_log WHERE scheduled_at < NOW() - INTERVAL '30 days'$$
);
```
