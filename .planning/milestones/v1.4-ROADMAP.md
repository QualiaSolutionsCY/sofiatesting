## Milestone Summary

**Key Decisions:**

- Use `import 'server-only'` for build-time enforcement (not runtime checks)
- Apply server-only guard to ALL modules accessing SUPABASE_SERVICE_ROLE_KEY
- Use throw Error() for server actions (not NextResponse)
- Verify chat ownership through RLS-compatible userId filtering
- Use safeParse() in API routes, parse() in server actions
- Use z.literal(true) for exact boolean confirmation
- CLI scripts retain minimal console.log (task name + completion only)

**Issues Resolved:**

- Service role key was accessible from client-side code (now build-time blocked)
- Server actions had no authentication checks (all 4 now protected)
- No input validation on API routes (Zod schemas on all critical endpoints)
- Debug console.log polluting production code (cleaned to zero)
- User and admin tables missing RLS write policies (now fully covered)

**Issues Deferred:**

- Schema/DB mismatch: 8 Drizzle tables don't exist in production
- Migration history: local/remote mismatch prevents supabase db push
- GET /api/admin/agents query params not Zod-validated (low risk)
- PATCH /api/vote uses manual validation instead of Zod (low risk)

**Technical Debt Incurred:**

- Drizzle schema needs sync with production DB (8 phantom tables)
- Migration history repair needed before supabase db push works
- upload_locks table has RLS enabled but no policies (intentional — service_role only)
